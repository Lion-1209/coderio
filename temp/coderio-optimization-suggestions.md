# coderio 优化建议

## 1. 架构层面
### 1.1 明确分层边界
当前架构总体是清晰的分层单体，但建议把“复用 seam”显式文档化：crew 复用 agent loop 时传 `harness=None`，这属于跨层特例，建议在接口注释或单独 seam 文件中标明，避免后续维护者误改。

### 1.2 显式化配置能力
`SkillsConfig.harness` 等字段虽然存在，但建议确保所有配置字段都有对应文档与默认值说明；若未来增加新开关，应同步更新 loader、bootstrap、README 与 TUI 展示。

## 2. 健壮性
### 2.1 加强 crew 持久化
crew 当前使用 `MemorySaver`，会话内可恢复，但进程外不可恢复。建议优先补齐 sqlite 持久化 checkpoint，尤其是长任务流水线。

### 2.2 会话写入健壮性
`Session.append` 直接追加 jsonl，若写入中途中断可能留下尾部不完整记录。建议引入原子写或写前校验机制，减少损坏会话文件概率。

## 3. 可观测性
### 3.1 结构化日志
当前 UI 反馈主要面向终端用户，建议增加结构化日志/事件流，便于后续调试、审计与自动化测试。

### 3.2 指标埋点
对 round 数、工具调用分布、harness 拦截次数、crew fix 循环次数等指标做轻量埋点，可帮助评估模型行为与工作流有效性。

## 4. 性能
### 4.1 Prompt 预算控制
已实现 30% 上下文预算告警，建议进一步细化：对长期运行的 session 做历史压缩或摘要，避免单轮 context 过度膨胀。

### 4.2 大历史场景
crew/session 在消息很多时可能变慢，建议在 resume 与 crew 状态传递时评估是否需要增量加载或摘要裁剪。

## 5. 工程体验
### 5.1 类型与校验
工具层、配置层已使用 pydantic，但仍有部分位置依赖运行时行为与字符串键。建议统一入口参数校验，减少隐式失败。

### 5.2 测试覆盖
已有较完整测试，建议补充 crew 在 auto 模式与交互模式的端到端覆盖，以及 harness 与 crew 同时启用的边界测试。

## 6. 可维护性
### 6.1 减少魔法字符串
`stage`、`writes`、`tool_name` 等存在多处字符串字面量，建议收敛为常量或枚举，减少拼写风险。

### 6.2 文档同步
README、`docs/coderio-architecture.md` 与代码实现要保持同步；建议在每次结构性变更时更新架构说明与已知限制部分。
